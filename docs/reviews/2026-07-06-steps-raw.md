# Review ‚Äî multi-step capture + allow_raise + nine resume/reset cases

Scope: `50dcaad` (capture.py steps form + allow_raise + fast tests) and `HEAD`
`bc368fb` (nine case JSONs). Reviewed the full current `harness/capture.py`,
`harness/run.py`, `harness/compare.py`, `tests/test_capture_validation.py`, and
all nine new cases. Cross-checked the reset/again/get_time/fact-clearing paths
against the oracle source at the pin (`oracle/pyreason/pyreason/pyreason.py`).
Fast tier green (`uv run pytest -m "not e2e"` ‚Üí 23 passed). The oracle env was
not run (a full run is in flight).

Overall the wrapping design is sound: `allow_raise` and step outcomes are always
wrapped (`{"raised": ...}`), and comparison is per-probe-id, so the
"wrapped `{"raised": false, "value": X}` vs bare `X`" collision the brief worried
about is **structurally impossible** ‚Äî `allow_raise` is a per-probe declaration
shared by both engines, so a given id is wrapped in both artifacts or neither.
Likewise the reset-family `get_time` no-program-vs-with-program distinction the
cases lean on is **source-accurate** at the pin. The findings below are the
residual gaps.

---

## High

### H1 ‚Äî A successful reason step banks only `{"raised": false}`; a reason step with no probe (or whose result is never re-probed) can mask a genuine reasoning divergence

`harness/capture.py:276-298` (`run_step`) and `:337-347` (steps loop).

`run_step` returns `{"raised": false}` for *any* op that does not raise ‚Äî the
record captures **whether** the op raised, never **what it computed**. The
reasoning effect is only observable through interp-consuming probes run *after*
the step. `_steps_fault` (`capture.py:80-120`) enforces that an interp probe be
preceded by a reason step, but it does **not** require a reason step's result to
be observed by any probe. So a case shape that passes `validate_case` ‚Äî

```json
{"steps": [
  {"id": "r1", "op": "reason", "args": {"timesteps": 2},
   "probes": [{"id": "t", "kind": "get_time"}]},
  {"id": "add", "op": "add_fact", "args": {"text": "popular(Justin)"}},
  {"id": "r2", "op": "reason", "args": {"timesteps": 2, "again": true}}
]}
```

banks `{"raised": false}` for `r2` in **both** engines even if engine A and
engine B derive completely different interpretations at that resume, because the
`get_time` probe only reads the clock and nothing reads `r2`'s interpretation.
`compare_probes` then reports no mismatch ‚Üí **pass**, a banked equivalence for
two genuinely different behaviors.

Failing scenario: the divergence is invisible whenever a reason step's derived
interpretation is not read by a subsequent interp probe on the *same* or a later
step. Note `get_time` and the step outcome are both too coarse to expose it.

Scoping (verified): **none of the shipped nine trigger this** ‚Äî every successful
reason step in the nine is followed by an interp probe that reads its result, or
is an expected-raise step (where the `raised` flag itself carries the signal).
So this is a latent validation hole, not a live wrong verdict. But it is exactly
"a case shape that passes `validate_case` but silently mislabels at run time"
and it *would* mask a divergence, hence High.

Suggested fix: in `_steps_fault`, require that a `reason` step either (a) declares
`allow_raise`-free interp probe(s) on itself or a later step that read its
interpretation, or (b) be explicitly marked as an expected-raise / effect-not-
observed step (e.g. `observe: false`), so a silently-unobserved successful reason
is an authoring fault rather than a vacuous `{"raised": false}`.

## Medium

### M1 ‚Äî A reason step swallows an authoring-typo kwarg as banked "engine behavior," converting a usage fault into a pass

`harness/capture.py:98-104` (step arg validation) and `:285-297` (`run_step`).

`_steps_fault` constrains args only for `add_fact` (must carry `text`) and the
reset family (must be empty); a `reason` step's `args` dict is passed straight to
`pr.reason(**args)` with **no** key validation. `run_step` then catches every
exception, including `TypeError: reason() got an unexpected keyword argument`,
and banks it as a raise-record. Contrast the one-step form (`capture.py:349-352`):
there `pr.reason(**inputs["reason"])` is *not* wrapped, so the same typo raises
out of `run_case` ‚Üí exit 1 ‚Üí judged **error** (visible, and correctly a non-
finding label). In the steps form the identical typo is banked as data; in
self-proof both engines raise the same `TypeError` ‚Üí **pass**. The case's stated
purpose (e.g. "continuation semantics") is never exercised, yet it reports green.

Failing scenario: `{"op": "reason", "args": {"timesteps": 1, "restrat": false}}`
(typo `restrat`) banks a `TypeError` outcome and passes, silently proving nothing
about restart.

Suggested fix: validate `reason`-step arg keys against a known set
(`timesteps`, `convergence_threshold`, `convergence_bound_threshold`, `queries`,
`again`, `restart`, ‚Ä¶) in `_steps_fault`, the same way `apply_settings`
(`capture.py:301-307`) refuses an unknown knob rather than silently testing a
default. An unknown reason kwarg should exit 2, not bank a raise.

### M2 ‚Äî An engine-introduced raise on a probe *without* `allow_raise` is judged `error`, not `divergent` ‚Äî a real cross-engine divergence mislabeled as harness failure

`harness/run.py:69-72` (the per-artifact error check runs first) with
`harness/capture.py:255-268` / `:399-405`.

A probe that raises without `allow_raise` propagates ‚Üí `run_case` exits 1 ‚Üí
`load_artifact` synthesizes `{"error": ...}` ‚Üí `judge_case` returns
`status: "error"` before it ever reaches `compare_probes`. The runner docstring
and charter define `error` as "a harness or environment failure and never a
claim about either engine." So if a rewrite engine *newly raises* on a probe the
author did not declare `allow_raise` for (precisely the "the rewrite diverges by
raising where the oracle didn't" case), the whole case is labeled `error` and an
operator following the taxonomy would treat it as a flaky harness/env problem ‚Äî
masking a genuine engine divergence behind the wrong label. This is inherent to
the design but is sharpened by the steps form, where raises are now first-class
observations for *declared* probes only.

Concrete exposure in the nine: `reason-again-restart-true` marks `nodes-after`
and `interp-after` with `allow_raise` but leaves `trace-after` (`rule_trace_node`)
and `time-after` (`get_time`) undeclared. If a rewrite makes `get_rule_trace`
raise on the restart-reconstructed trace, that divergence lands as `error`, not
`divergent`.

Suggested fix: in `judge_case`, distinguish a capture that failed on a *probe*
raise (engine behavior) from a genuine harness/usage failure ‚Äî e.g. have capture
tag probe-raise capture failures in the artifact `error` payload, and have the
judge route a probe-raise failure that differs across engines to `divergent`.
At minimum, document that any bare (non-`allow_raise`) probe raise is judged
`error`, so authors add `allow_raise` on every probe that *could* raise under a
candidate rewrite.

### M3 ‚Äî `reset-settings-restore` purpose overstates what is observed: the second run is fact-less, so its row-level claim ("'Occurred Due To' falls back to rule names") is never exercised

`harness/cases/reset-settings-restore.json:3` (purpose) and `:40-44`
(`second-reason`). Verified against oracle source: `_reason` sets
`__node_facts = None`/`__edge_facts = None` on exit (`pyreason.py:1623-1624`),
and a fresh `_reason` re-inits them to *empty* (`pyreason.py:1553-1556`,
`if __node_facts is None: ‚Ä¶ empty_list`). `reset_settings` does not restore facts.

Consequence: the `second-reason` step runs with the `popular(Mary)` seed gone
(only graph-attribute edge facts remain). The rule
`popular(x) <-1 popular(y), Friends(x,y), owns(y,z), owns(x,z)` needs a
`popular(y)` seed, so it never fires and the second node trace has **no rule-
firing rows**. The purpose's specific row-level claim ‚Äî "'Occurred Due To' falls
back to rule names" ‚Äî cannot be observed with zero firing rows. The intended
isolation (atom_trace on ‚Üí off via `reset_settings`) is confounded with the fact-
clearing: the trace changes for *two* reasons at once.

Nuance (in the case's favor): the atom_trace **column** set still differs by
construction even with zero rows, so a rewrite whose `reset_settings` failed to
restore `atom_trace` *would* still surface as a divergence via the columns. So
this is an overstated-purpose / mis-attributed-observation fault, not a coverage
hole ‚Äî but the `surface_items` "setting:atom_trace" restore claim and the row
narrative are not honestly earned.

Suggested fix: either re-add `popular(Mary)` via an `add_fact` step before
`second-reason` so the rule fires under defaults and the row-level claim is real,
or rewrite the purpose to claim only the column-shape restore that is actually
observed.

## Low

### L1 ‚Äî `allow_raise` defeats the "missing constructor / binding fault" protection

`harness/capture.py:255-268` (`run_probe_recorded`) vs `:210-231`
(`probe_expect_raise`, whose docstring promises "a missing constructor is a
harness/binding fault and propagates as a capture failure rather than wearing the
engine-behavior label"). `run_probe_recorded` catches **every** `Exception` when
`allow_raise` is set, so an `expect_raise` probe (or any probe) marked
`allow_raise: true` whose underlying attribute/constructor is missing banks the
`AttributeError` as an engine observation instead of failing the capture. The
escape hatch cannot tell a null-interp `AttributeError` (intended data) from a
missing-method `AttributeError` (binding fault). Latent: no `allow_raise`
`expect_raise` probe exists in the nine, and both real engines carry the
constructors. Fix: narrow the `allow_raise` catch to exceptions raised from
*inside* the engine call, or exclude `AttributeError` on the probe entry point.

### L2 ‚Äî `reason-again-no-program` purpose claims equivalence to a fresh run that the harness never computes

`harness/cases/reason-again-no-program.json:3`. Purpose: the again-first-call
short-circuit is "proven by outputs matching a fresh run's shape." The harness
compares this case oracle-vs-oracle only; there is no sibling fresh-run case whose
probe outputs are diffed against it, so "matching a fresh run" is asserted, never
checked. The case genuinely proves oracle‚â°oracle on the again-first path, which
is weaker than the stated claim. Fix: reword the purpose to what is verified, or
add a paired fresh-run case and a documented expectation that their probe outputs
coincide.

### L3 ‚Äî `test_empty_or_malformed_steps_are_rejected` "rest" assertion is trivially satisfied

`tests/test_capture_validation.py:160`. The unknown-op case uses `op: "rest"` and
asserts `"rest" in validate_case(...)`. The fault message embeds the full
`STEP_OPS` list, which contains `reset`, `reset_rules`, `reset_settings` ‚Äî every
one a superstring of `"rest"`. So the assertion would pass even if the offending
op name were dropped from the message. The docstring claims the unknown op is
"named"; the assertion does not actually prove that. Fix: assert on the quoted op
token, e.g. `"'rest'" in ...`, or assert the op name appears outside the ops list.

## Checked and clear (not defects)

- **Wrapped-vs-bare collision** (brief's concern 2): impossible ‚Äî `allow_raise`
  is per-probe, identical across engines; a given id is wrapped in both artifacts
  or neither. Step outcome `{"raised": false}`, wrapped success
  `{"raised": false, "value": X}`, and `expect_raise` `{"raised": false, "parse": ‚Ä¶}`
  are key-distinct and each compared only against the same id.
- **Step ids in the compared set**: `run_case` writes `probes[step["id"]]` and
  `digests` covers it (`capture.py:343,369`), so step outcomes are both
  reproducibility-gated and divergence-compared. Judge taxonomy holds for steps
  artifacts (step ids are ordinary probe-map keys).
- **id namespace collisions**: `_steps_fault` unions step ids and probe ids into
  one uniqueness check (`capture.py:116-119`); `run.py:135-146` rejects duplicate
  *case* ids before capture. Both correct.
- **reset-family `get_time`**: verified against the pin ‚Äî `get_interpretation`
  raises when `__program is None` (caught ‚Üí 0) but returns a null `interp` when a
  program exists post-reset (uncaught `AttributeError`). The nine model the
  no-program (bare `get_time`) vs with-program (`allow_raise: true`) split
  correctly, as do `reset-rules-*` (interp survives `reset_rules` ‚Üí `get_time`
  answers).
- **`reason-bare-again-no-facts`**: the bare-resume `TypeError` from
  `numba.typed.List(None)` is banked as the step outcome (`capture.py:296-297`),
  and the trailing `time-after` `get_time` reads the still-live first-run interp
  (the raise is inside `_reason_again` before reassignment), so it does not raise.
  Correct.
