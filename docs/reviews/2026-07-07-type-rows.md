## Findings

### M1 (evidence-discrimination) — clause-level threshold gating was claimed beyond what the cases pin — FIXED

The board's type:Threshold note claimed "a number threshold gates at the
CLAUSE level — the qualifying-grounding count across the whole clause, not
per head grounding", citing the `-two`/`-default` twins. But in that pair's
graph (A—B, A—C, one qualifier `popular(B)`, single head candidate A) the
per-head count and the whole-clause count are both 1 — the twins cannot
tell the two gating semantics apart; both predict the same outcome. This is
the session-12/13 failure shape again: a source-reading promoted to a
behavioral "not X" without a discriminating observation.

The claim itself is **true** — confirmed live with a discriminating screen
(two disconnected Friends pairs A—B / C—D, `popular(B)` + `popular(D)`,
`greater_equal number/total 2` on the popular clause: each head's
neighborhood holds ONE qualifier, the clause's whole-graph grounding holds
2, and `trendy(A)` **and** `trendy(C)` both derive at t=1). Fix: that shape
is now banked as **`threshold-number-gate-clause-level`** (corpus 93 → 94,
exact compare, targeted rerun ALL PASS) and the board note names the
discrimination explicitly instead of leaning on the non-discriminating
twins.

### M2 (engine-env kernel cache) — the registrand poisoning recurs on every registry capture, and the author's scoping was refuted live — FIXED + repaired

The author hit and repaired a mid-session fault (scratchpad-module njit
registrands poisoning numba's on-disk cache index) and left a standing
note: committed-module registrands write index entries referencing
`harness.reference_fns`, "importable in every repo-rooted run", risky only
for processes "started OUTSIDE the repo". Audit findings:

1. **The poisoning is not a one-time event — it recurs on every registrand
   capture.** After the author's session and my pre-fix rerun, six kernels'
   cache indexes in `oracle-env/.../pyreason/cache/` again carried
   `harness.reference_fns` references (`Interpretation.reason`, `annotate`,
   `_call_head_function`, `_determine_node_head_vars`,
   `_determine_edge_head_vars`, `_ground_rule` — the same six the author
   repaired).
2. **The "repo-rooted" scoping is wrong: sys.path, not cwd, decides.**
   Reproduced live: a plain-script screen run from the repo root
   (`oracle-env/bin/python <script>`) died at numba's `_load_index` with
   `ModuleNotFoundError: No module named 'harness'` — script invocation
   puts the script's directory, not the cwd, on `sys.path`. Only `-m`-style
   repo-rooted invocation (the runner's shape) survives. The failure is a
   hard error, not self-healing.
3. **The author's compile-cost claim ("compiled once, then disk-cached
   across processes; repeats ran warm") is refuted by observation.**
   Dispatcher-bearing signatures never hit the cache (a fresh process's
   `resolve()` builds new dispatcher objects, so the pickled signature key
   never matches): during my rerun, each registrand capture appended a NEW
   ~10 MB `.nbc` overload of `Interpretation.reason` (overloads 25, 26, 27
   written in one case's four captures). Every registrand capture pays the
   full compile *and* grows site-packages — a full-corpus sweep would add
   on the order of 100 MB per run, forever.

Fix (in `harness/capture.py`, seam-tested): a capture whose case applies a
callable-registering op (either spelling — `case_registers_functions`)
snapshots the engine env's bundled kernel-cache before the engine imports
(file set + index bytes; the cache directory is located via
`importlib.util.find_spec`, without initializing the engine) and restores
it in a `finally` after the capture: files the capture added are removed
and rewritten indexes get their prior bytes back, leaving the bundled cache
exactly as a registrand-free run would. Data files are add-only (each new
specialization gets a fresh counter-named `.nbc`), so the few-KB indexes
are the only bytes snapshotted. A simpler-looking alternative —
re-pointing `NUMBA_CACHE_DIR` at a per-capture directory — was tried first
and **refuted by screen**: the engine package's `__init__` re-points the
variable before its kernels are decorated, numba's cache locators bake
`config.CACHE_DIR` in at decoration time, and the compiler re-reads the
environment on every compilation (numba `core/compiler.py:400` calls
`reload_config()`), so the override left the per-capture directory empty
and the bundled cache poisoned; the finding is recorded in the mechanism's
docstring. Residual exposure: a capture killed mid-run skips the restore —
registrand cases have no crashing arms, and the repair recipe below covers
any residue.

Repair: deleted the six poisoned kernels' cache sets from the oracle-env
virtualenv (index + code files, 325 → 219 files; zero indexes reference the
registrand module afterward — the same runtime-cache maintenance the author
performed mid-session, in the disposable env, never the pinned tree), then
re-verified hello-world and the registrand cases ALL PASS under the
restore mechanism and that the bundled cache carries **no**
`harness.reference_fns` references and no new overload files after those
runs.

Standing operational note (re-scoped): screens and tools must run the
engine either via `-m` from the repo root or against a clean cache; njit of
non-committed modules in the oracle env remains forbidden (the author's
rule stands). With the confinement in place, harness captures no longer
create the exposure.

## Mechanism review (commit 0aa8d6e)

- **Validation parity** — registry names, rule specs (list/dict
  custom_thresholds, weights), query-text lists, and interval/label probe
  specs all validate identically in both case spellings via the shared
  helpers (`_apply_fault`, `_rule_spec_fault`, `_reason_args_fault`); every
  authoring fault exits 2 before the engine imports. Consistent with the
  session-13 conventions.
- **Raise-attribution discipline** — registrand resolution, Query
  construction, and the engine-binding getattr all sit outside the
  recording try in both forms; construction raises are capture failures,
  never banked behavior. `interval_probe`/`label_probe` refuse
  `allow_raise` with case-specific rationale (the label probe records its
  one raising relation itself, keeping the probe total).
- **Additivity** — verified in the diff: a threshold-less rule spec calls
  `pr.Rule(text, name, infer_edges)` with the pre-existing positional
  shape and no kwargs (seam-tested: "no smuggled kwargs on the plain
  shape"); `build_reason_args` returns the identical dict object when no
  `queries` key rides. No pre-existing case's capture call shape changes,
  so skipping the corpus sweep on additivity grounds was sound (the
  Phase-3-boundary session runs everything anyway).
- **Fingerprints** — the query branch renders all four parsed fields +
  echoed text; the threshold branch the three stored attributes +
  `to_tuple()`; `canonical()` reduces the parser's Interval bound via its
  lower/upper branch. Misparses and unvalidated `thresh` values bank
  verbatim, as designed.
- **Registry** — stdlib-only import and `__name__`-keying held by
  fast-tier tests; njit laziness means the deliberately-wrong-shape stubs
  never compile; all five reference functions are consumed by committed
  cases (no dead scaffolding). The `numba = None` module-global pattern for
  the head function's `numba.typed.List` return is documented and bound in
  `resolve()` before first call.
