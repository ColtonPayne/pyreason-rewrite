# The AC-4 workload ladder — rungs, rationale, constraints

The committed performance workload set for Phase 4 (the execution-layer
evidence path). Three rungs — small / medium / large — each a **committed
harness case** (`harness/cases/perf-ladder-{small,medium,large}.json`,
provenance `perf ladder`), so performance and equivalence share inputs: the
same case files run through the differential harness for the equivalence
verdict and through the benchmark runner (`harness/bench.py`) for the
numbers. The committed ladder precedes any floor verdict; thresholds beyond
the parity floor are proposed only after baselines exist (banked beside this
file as `oracle-baselines.md`).

## What the engine actually spends time on

The ladder scales the dimensions the pinned engine's reasoning loop pays
for, per timestep of the temporal loop:

- **Grounding** — matching each rule's clauses against the graph: node
  clauses scan node worlds, edge clauses scan edge label tables; a rule with
  a multi-variable join (hello-world's 4-clause shape) grounds over
  node-neighborhood products, the superlinear term.
- **Rule firing / interval updates** — each satisfied grounding schedules a
  head update; bound changes feed the next timestep's re-derivation.
- **Fact application** — facts with time windows assert bounds per timestep
  while active; staggered windows keep state *changing* across the run
  (default `persistent=False` lets expired bounds revert), so later
  timesteps do real work instead of idling at a fixpoint.

## The rungs

Scaling dimensions per rung: graph size (nodes/edges), rule count and
chaining depth, timesteps, fact volume. Every fixture is generated by the
committed, seeded, stdlib-only generator `tools/gen_perf_fixtures.py`
(ring backbone for connectivity + seeded random extra edges; every edge
carries exactly one of `rel0`/`rel1`/`rel2` with value 1). Regeneration is
byte-identical; the script errors if a committed fixture drifts from its
seed. No randomness exists at measure time — fixtures and cases are
committed artifacts.

| rung | nodes | edges | rules (chain depth) | timesteps | facts | runtime class |
|---|---|---|---|---|---|---|
| small | 50 | 150 | 2 (seed→t1→t2) | 10 | 5 | smoke |
| medium | 400 | 2,400 | 3 (seed→…→t3) | 40 | 25 | standard |
| large | 1,000 | 6,000 | 5 — 4-deep chain (seed→…→t4) **plus** the 4-clause join `t5(x) <-1 t1(y), rel0(x,y), rel1(y,z), rel2(x,z)` | 75 | 60 | long |

Rule texts are a tier chain (`t1(x) <-1 seed(y), rel0(x,y)`, then
`t2 <- t1`, `t3 <- t2`, `t4 <- t3` over cycling relation labels); `seed` is
asserted only by facts, staggered `start = i mod (timesteps/2)`,
`end = timesteps`. The large rung's join rule reuses the proven hello-world
clause shape (node clause + three edge clauses over three variables) at
scale — it is the grounding stressor and dominates large-rung cost in both
engines. Probes per case: `filter_sort_nodes` on two chain labels (both
ends of the chain, or the deep end + the join) plus `get_time` — a
per-timestep, per-node bounds fingerprint, strong enough for equivalence,
bounded enough to keep artifacts in the single-digit-MB range.

### Why these sizes

- **small** — hello-world order of magnitude, one decade up in nodes. Sized
  to be dominated by fixed per-run cost: it is the **cold-start rung**
  (fresh-process import + first `reason()`), and its reasoning number is the
  floor any engine pays before scale matters. Screened single-run capture
  ≈ 6.4 s oracle / 0.2 s rewrite wall-clock.
- **medium** — one decade up in edge-timestep work (2,400 edges × 40 steps
  vs 150 × 10), chain depth 3, no join rule: the *linear-scaling* rung.
  Screens: oracle `reason()` ≈ 3.3 s (barely above its ≈ 2.7 s small-rung
  floor — the pinned engine's per-call fixed overhead swamps chain-rule
  grounding at this scale; that flatness is itself baseline evidence),
  rewrite ≈ 0.6 s.
- **large** — another decade of edge-timestep work **plus** the join rule
  and depth 5: stresses grounding at scale while staying laptop-local and
  intentionally lightweight. Screens: oracle `reason()` ≈ 17.1 s, rewrite
  ≈ 17.6 s, single capture ≈ 22 s — comfortably inside the ≤ ~3 min
  single-run budget, far above measurement noise.

Screened wall-clocks above are single smoke runs on the campaign machine
(2026-07-12, oracle-env numba cache warm); the banked medians with noise
bands live beside this file in `oracle-baselines.md`.

## Structural constraints (not tribal knowledge)

- **B16 — bounded timesteps everywhere.** The pin's fp schedule cannot
  terminate on `timesteps=-1` (its only exit is `t == tmax`,
  `interpretation_fp.py:272-273` — a hang, not a raise; operator-adjudicated
  batch item B16). **No ladder rung may combine `fp_version=True` with
  `timesteps=-1`.** The ladder makes this structural twice over: every rung
  declares an explicit positive `timesteps`, and no rung sets `fp_version`
  at all. Any future fp-schedule rung must keep an explicit bound.
- **B34 — `output_to_file` stays off.** The pin rebinds `sys.stdout` to the
  redirect file and never restores it (batch item B34); a timing run must
  never carry that knob. No rung sets it (harness validation would also
  refuse it without an `output_file` probe).
- **Default engine config.** No `fp_version`, no `parallel_computing`, no
  `output_to_file`; `verbose=false` (the spine-case convention — per-step
  console printing is run schedule, not reasoning work). Everything else at
  pinned defaults. The baseline ladder measures the deterministic public
  surface's default path; knob-position ladders would be new committed
  rungs, not edits to these.

## Reproduction

```
uv run python tools/gen_perf_fixtures.py            # verify fixtures match their seeds
PYTHONHASHSEED=0 uv run python -m harness.run \
  --cases harness/cases/perf-ladder-small.json \
  --engine-a oracle-env/bin/python --engine-b scripts/rewrite-python \
  --results results-phase4-ladder                    # equivalence, one rung (repeat per rung)
```
