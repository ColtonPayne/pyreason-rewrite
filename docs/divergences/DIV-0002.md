# DIV-0002 — non-string IPL entries: the pin raises an address-derived ValueError, the rewrite a stable one

- id: DIV-0002
- case ids: none — the input is un-bankable (the pinned engine cannot
  reproduce its own observable: the raise MESSAGE is address-derived, so
  the harness scores the case `irreproducible (engine-a)` before any
  cross-engine compare — screened 2× through the harness, slice-8 review).
  Nearest committed case: ipl-load-malformed (all four of its arms are
  deterministic plain-Python raises; none reaches the typed-container
  append).
- surface items: fn:load_inconsistent_predicate_list
- observed (input: an IPL YAML whose pair entries are YAML integers —
  `ipl:\n  - [1, 2]\n`):
  - **oracle (pin e1a94af33e1f):** `parse_ipl` constructs plain
    `Label(1)`/`Label(2)` without complaint (the pure-Python constructor
    checks nothing), but the append into the numba
    `typed.List(Tuple((label_type, label_type)))` unboxes each Label's
    `_value` as a numba string (label_type.py:86-94 `unbox_label` →
    `c.unbox(types.string, ...)`), and unboxing an int as a unicode
    struct raises `builtins.ValueError` whose message reads garbage
    memory as a code point: 4 fresh engine processes gave 4 distinct
    messages, all matching `character U+[0-9a-f]+ is not in range
    [U+0000; U+10ffff]` (observed: U+1552710, U+51c2710, U+341e710,
    U+5c6e710). Raise TYPE stable, message unstable — same-engine
    repeats compare unequal, so no exact-compare case can bank the arm.
  - **rewrite (adjudicated behavior, implemented session 25):** the
    append seam raises `builtins.ValueError` — same seam and type as the
    pin — with the stable message `IPL entries must be strings; got
    int: 1` (the `ipl_pair` guard, src/pyreason/_state.py; the loader
    calls it at src/pyreason/_loaders.py::load_inconsistent_predicate_list,
    and the raise leaves the previously-held IPL untouched, the pin's
    parse-then-assign shape). As filed (sessions 23-24, pre-adjudication)
    the plain-list append accepted the pair — the divergence this record
    was opened on. No committed case reaches the arm; every committed IPL
    fixture carries string entries.
- first-seen session: 23 (phase-3 slice-8 review probe
  `rv-ipl-nonstring-pair`; the committed slice's 4 cases and 4 seam
  tests are unaffected — string entries everywhere).
- classification: discovered-divergence as filed — a pinned-container
  side effect (unbox type enforcement) the rewrite's plain-list
  transcription did not carry; not intentional, not absorbable.
  Adjudicated 2026-07-11: the type enforcement is now reproduced (the
  `ipl_pair` guard), and the residual message-text difference stands as
  documented, tested intentional behavior per AC-6.
- reproducer: tests/test_div_0002_reproducer.py (e2e-marked: subprocess
  run of the pinned engine on a generated non-string IPL file, asserts
  the ValueError type + the character-not-in-range message shape) +
  tests/test_rewrite_state_loaders.py::
  test_ipl_yaml_nonstring_entries_raise_stable_valueerror and
  ::test_add_inconsistent_predicate_nonstring_raises_the_same_guard
  (fast tier: the adjudicated guarded raise at both IPL entry points;
  before adjudication this slot named the provisional-acceptance test
  the guard replaced).
- status: adjudicated
- provisional behavior (superseded by the adjudicated behavior below;
  kept as the record of the fork): pre-adjudication the rewrite kept the
  acceptance (zero validation, the literal parse_ipl transcription),
  because the fork is consequential and neither arm can match the pin
  exactly: (a) guard the loader — raise `builtins.ValueError` on
  non-string entries at the append seam; same type and seam as the pin,
  but the message must be invented (the pin's own text is unreproducible
  even pin-vs-pin), or (b) keep the acceptance and document that
  non-string labels fail later and differently (in the rewrite a
  non-string Label breaks at first `str()`/`hash()` consumption inside
  reason()).
- adjudicated behavior: option (a). Non-string entries raise
  `builtins.ValueError` at the append seam with the message
  `IPL entries must be strings; got {type}: {value!r}` (e.g. `IPL
  entries must be strings; got int: 1`). The text is **stable** — it is
  built only from the entry's type name and repr, both deterministic for
  every value YAML safe_load can produce (str/int/float/bool/None/
  list/dict/date), with no addresses or environment paths — and
  **honest**: it names the offending entry and states the actual
  contract (IPL labels must be strings, because the pinned typed-list
  unbox reads each Label's `_value` as a numba string). It deliberately
  does NOT imitate the pin's character-code text, which reads garbage
  memory and is unreproducible by construction. The guard fires at the
  same logical point the pin's unbox fails — per pair, inside the parse
  loop, before the rebind — and only for the entries the pin's unbox
  would reject (any non-str `_value`; str entries pass through
  unchanged). Guard parity: the guard lives at the shared choke point
  (`ipl_pair`, src/pyreason/_state.py), so `add_inconsistent_predicate`
  — whose pinned twin appends into the same typed list
  (pyreason.py:629) and fails the identical unbox — raises the same
  ValueError. Seam tests: tests/test_rewrite_state_loaders.py::
  test_ipl_yaml_nonstring_entries_raise_stable_valueerror and
  ::test_add_inconsistent_predicate_nonstring_raises_the_same_guard.
- verdict: **option (a), accepted by the operator 2026-07-11 (adjudicating
  session: 25, Phase-3 adjudication batch item A2)** — guard the loader:
  raise `builtins.ValueError` on non-string entries at the append seam,
  the same type and seam as the pin, with a stable, honest message (the
  pin's own message is address-derived and unreproducible even pin-vs-pin
  — that fact lives in this record; the message difference is unpinnable
  by construction). This follows the typed-container-semantics precedent
  (slice-6 TypedRuleList, slice-7 TypedComponentDict both model pinned
  container raise shapes). Implemented by session 25; the guarded
  behavior, the chosen message text, and its rationale are recorded in
  the observed/behavior fields above.
